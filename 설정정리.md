### naver_check.py 핵심 로직 (2026-06-23 디버깅으로 확정, 메디안과 동일)
- 네이버 **모바일 통합검색**에서 발행URL이 검색결과(인기글 등)에 있으면 '노출중'

## 캡처 이미지 수동 교체 기능 (2026-07-07, 커밋 977ec95, prod 배포완료)
- 캡처 탭 프리뷰 모달 **이미지 우측 하단 "이미지 교체" 버튼** → 안내 패널(권장 사이즈: 기본 390×844px / 전체보기 가로 390px, PNG·JPG·WEBP 4MB 이하, 되돌리기 불가 경고) → 파일 선택 즉시 교체
- API: `POST /api/captures/replace` (multipart: id, mode=basic|full, file)
  - Storage `amos-captures`에 `captures/{date}/{post_id}[_full]_edited_{ts}.{ext}` 새 경로로 업로드(CDN 캐시 회피) 후 `amos_daily_captures.image_url`/`full_image_url` 갱신
  - 검증: 파일 타입/4MB 제한, 존재하는 캡처 레코드만 교체 가능
- 현재 화면에 보이는 모드(기본/전체보기)가 교체 대상
- ⚠️ 당일 캡처는 다음 cron(08/14/20시)에 노출 재확인되면 자동 캡처로 다시 덮일 수 있음 (지난 날짜는 안전)

## 관리자패널 컬럼 개편 + 조회수 누적통합 (2026-07-07, 커밋 92a1654·a8d50e7 prod 배포)
- 컬럼: `# | 브랜드 | 상태 | 진행 | 제품 | 구분 | 구분2 | 키워드 | 검색량 | 노출탭 | 발행URL | 이미지호스팅URL | 제품링크URL | 총노출일 | 조회수 | 총조회수 | 총클릭수`
- 구분(category)=드롭다운 건바이건/월보장, 구분2(category2)=자유입력, 진행(progress)=배지 클릭 작업중↔작업완료
- 검색량: `/api/search-volume` + `lib/naver-searchad.ts` — 네이버 검색광고 keywordstool, 오늘 조회 기준 최근 30일 월간검색량. ★HMAC 서명은 마침표 구분 `timestamp.GET.uri` (newline은 403). env NAVER_API_KEY/NAVER_API_SECRET/NAVER_API_CUSTOMER_ID (Vercel 등록완료)
- 조회수(통합): `lib/combined-views.ts` — `combined = views_base + max(0, 현재소스 - views_offset)`. 카페URL=카페API(cafe_views), 블로그URL=이미지호스팅 조회수(image_views). 발행URL 교체 시 PATCH가 옛 URL 라이브 최종조회→views_base 스냅샷(리셋 없이 누적). 같은 이미지 재사용 시 offset으로 중복방지
- DB 추가컬럼: category, category2, progress(default '작업중'), views_base(int 0), views_offset(int 0) — Supabase PAT Management API로 DDL 실행
- 엑셀 export: 총노출일|조회수|총조회수|총클릭수 컬럼 추가 (총클릭수=hwaseon-url /api/stats)
- 라이브 e2e 검증완료: 검색량 27키워드 실수치(해외IP OK), 진행토글, 스냅샷(1000 누적), 엑셀 헤더
- ⚠️ amos_posts.product는 실DB NOT NULL — API로 product 없이 POST하면 500

## 캡처 헤더 테두리 버그 4차 근본수정 (2026-07-07, amos 3a4490b / median 14ee6a4)
- 증상(4번째 재발): 빨간 테두리가 매칭 글이 아니라 검색창+탭 헤더에 그려짐 (07-07 케라스타즈 시몽·두피케어)
- 근본원인: `_find_post_element`의 헤더 판별이 화면 구조 셀렉터(input/[role=tablist])에만 의존 → 네이버가 SERP 구조를 수시로 변경·A/B 실험해서 다른 변형이 서빙되면 가드가 뚫림. 1~3차 수정이 전부 "네이버 화면 구조 가정" 위에 있어 재발 반복
- 수정: 구조 무관 **위치 기반 가드** — ①매칭 링크보다 1.3×뷰포트 위에서 시작하는 블록 거부 ②검색폼/탭 하단선(~121px) 위쪽 블록 거부 ③카드 못 찾으면 매칭 링크 자체 반환(헤더 불가능)

- 교훈: 캡처 수정은 노출 키워드 실물캡처 검증 없이 완료 선언 금지
- 별개 이슈(코드 아님): 노출 됐다/안됐다가 사용자 화면과 어긋나는 건 GitHub Actions 미국 IP가 받는 네이버 검색결과가 한국과 순간순간 다르기 때문(추정, 한국IP에선 매칭 100% 정상 실측). 근본 해결 = 캡처를 한국 IP 환경으로 이전(amos·median 서버 분리 건과 연계 가능, 미착수)
